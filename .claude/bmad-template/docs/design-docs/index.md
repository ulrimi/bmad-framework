# Design Documents

This directory holds design documents for {{PROJECT_NAME}}.

## Purpose

Design docs capture the **why** behind significant technical decisions — the alternatives considered, trade-offs evaluated, and constraints that shaped the chosen approach. They serve as durable context for agents and engineers who encounter the code later.

## When to Write a Design Doc

- New subsystem or major feature with multiple viable approaches
- Significant refactoring that changes architectural boundaries
- Technology choices (libraries, databases, protocols)
- Cross-cutting changes that affect multiple modules

## Creating a New Design Doc

1. Create a new `.md` file named descriptively: `YYYY-MM-DD-short-description.md`
2. Include sections: Context, Decision, Alternatives Considered, Trade-offs
3. Add an entry to this index
